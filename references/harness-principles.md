# 双 Agent 框架原则

## 核心思想

- 制定标准和评判标准的应该是同一个 agent（规划/评审合一）
- 产出内容的 agent 不评判自己的输出（执行 agent 不自评）
- 用 markdown artifact 作为长流程的工作记忆
- 按具体标准评审，而非泛泛的质量判断
- 精确路由：优化结构（improve）vs 补搜信息（search）vs 被阻塞（blocked），三者不混淆
- 信息获取受阻时停下来，不假装可以通过改写文字解决

## 为什么这样设计

### 自评不可靠

Agent 倾向于对自己的输出过于宽容，即使质量明显一般也会自信地表扬。这不是偶尔的失误，而是系统性的偏差。让生成者批评自己的输出，得到的反馈几乎总是比独立评审者更温和。

### 分离评审更容易调优

当评审者是独立的，可以单独优化它的评审标准、prompt 和行为，而不影响生成流程。比让生成者"更严格地自我批评"有效得多。

### 规划者最适合做评审者

规划/评审 agent 在流程前期定义了"好报告长什么样"（Research Brief、Search Plan）。评审时它不需要重新理解别人的意图——它就是定义意图的那个 agent。这比让一个全新的评审者从零理解标准更高效、更一致。

### 通过文件通信确保可恢复

Agent 之间通过 markdown artifact（而非对话历史）通信。这意味着：
- 会话中断后可以恢复
- 每一步的决策都有可检查的记录
- 长流程不会因为上下文窗口限制而丢失关键信息

### 平台期检测防止无效迭代

连续 2 轮评审提升 < 0.7 分且主要弱点是信息获取时，自动停止。不会陷入"改一点文字、再评审、再改一点"的无效循环。

## 为什么长流程会漂移

### 推理链没有外化

上下文变长后，agent 的注意力分散。如果中间推理没有保存到 artifact 中，后续步骤会逐渐偏离最初的规划。Markdown artifact 链就是对抗这个问题的机制。

### 执行 agent 对自己的输出越来越宽容

这是自评偏差在长流程中的放大效应。随着已投入的"工作量"增加，agent 越来越不愿意承认产出质量不够。独立评审者没有这个包袱。

### 缺失信息被当作写作问题

当某个章节薄弱时，执行 agent 的本能反应是"改写得更好"，而不是"我缺少数据"。精确路由（improve vs search vs blocked）就是为了阻止这种错误归因。
